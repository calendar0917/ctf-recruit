# 动态实例设计

## 目标

为 Web/Pwn 等题目提供按用户分配的独立运行环境，并保证实例可以自动回收。

## 当前实现阶段

当前仓库已经实现动态实例的数据库驱动版基础闭环：

- API 接口已接入实例服务
- 默认按 `用户 + 题目` 唯一实例运行
- 服务内置到期实例 sweeper
- 题目运行配置从 PostgreSQL 读取
- 实例元数据写入 PostgreSQL
- 底层运行时通过 Docker Engine HTTP API 管理容器

当前能力边界如下：

- API 重启后，实例元数据不会丢失
- 示例动态题 `web-welcome` 已通过迁移写入数据库
- 当前还没有做 API 启动时的容器状态恢复
- 当前鉴权仍然用 `X-User-ID` 请求头占位

## 默认模型

- 分配模式：按用户独立实例
- 限制：同一用户在同一道题上最多一个运行中实例
- TTL：默认 30 分钟
- 暴露方式：宿主机端口映射
- 管理方式：API 服务直接管理 Docker Engine

## 生命周期

1. 管理员为题目配置运行镜像、容器端口、资源限制和 TTL
2. 选手点击“启动实例”
3. API 从数据库读取题目运行配置
4. API 检查该用户在该题上是否已有运行中实例
5. 若无实例，API 创建容器、分配端口、写入实例记录
6. API 返回访问地址和过期时间
7. 用户可查看实例状态，必要时手动销毁
8. 后台 sweeper 定期扫描数据库中的过期实例并停止、删除容器

## 当前接口约定

- 实例相关接口当前通过 `X-User-ID` 请求头模拟已登录用户
- 这只是开发阶段占位，后续会替换成 JWT 鉴权上下文

## 数据结构

### `challenge_runtime_configs`

保存题目级运行配置，包括：

- 镜像名
- 暴露端口
- 暴露协议
- 默认 TTL
- CPU / 内存限制
- 额外环境变量
- 启动命令覆盖

### `challenge_instances`

保存用户实例元数据，包括：

- 用户 ID
- 题目 ID
- 容器 ID
- 映射端口
- 状态
- 启动时间
- 过期时间
- 停止时间

## 安全边界

- 不使用 privileged 容器
- 默认限制内存和 CPU
- 默认只暴露题目需要的端口
- 用 Docker label 标记 `contest`、`challenge`、`user`、`instance`
- 实例过期后必须自动清理
- 管理员可以看到异常实例并手动回收

## 当前剩余缺口

1. API 启动时恢复数据库记录和真实容器状态
2. 把 `X-User-ID` 占位鉴权替换成 JWT
3. 提供管理端实例查询与强制终止接口
4. 增加题目运行配置的后台管理接口

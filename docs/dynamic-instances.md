# 动态实例设计

## 目标

为 Web / Pwn / Misc 等题目提供按用户分配的独立运行环境，并保证实例可以被限制、回收、审计和排障。

## 当前实现状态

当前仓库已经实现动态实例的数据库驱动版基础闭环：

- 实例相关 API 已接入 Bearer Token 鉴权
- 默认按 `用户 + 题目` 唯一实例运行
- 服务内置到期实例 sweeper
- API 启动时会先执行一次基础对账
- 定时任务会扫描数据库记录与 Docker 受管容器并纠正明显不一致状态
- 题目运行配置从 PostgreSQL 读取
- 实例元数据写入 PostgreSQL
- 底层运行时通过 Docker Engine HTTP API 管理容器

## 当前能力边界

- API 重启后，实例元数据不会丢失
- 示例动态题 `web-welcome` 已通过迁移写入数据库
- 当前访问方式仍以宿主机端口映射为基线
- 当前已经能处理“数据库有记录但容器已消失”和“Docker 里有受管容器但数据库没有记录”的基础场景
- 当前还没有做按题目或按用户的全局实例配额与冷却控制

## 默认模型

- 分配模式：按用户独立实例
- 限制：同一用户在同一道题上最多一个运行中实例
- 生命周期：题目级 TTL，默认仍按 `30m` 理解
- 暴露方式：宿主机端口映射
- 管理方式：API 服务中的 runtime 模块直接管理 Docker Engine

## 生命周期

1. 管理员为题目配置镜像、端口、资源限制和 TTL
2. 选手点击“启动实例”
3. API 从数据库读取题目运行配置
4. API 检查该用户在该题上是否已有运行中实例
5. 若无实例，API 创建容器、分配端口、写入实例记录
6. API 返回访问地址和过期时间
7. 用户可查看实例状态、续期或主动删除
8. 后台 sweeper 定期扫描数据库中的过期实例并停止容器、更新记录
9. 对账任务会终止失联记录，并清理 Docker 中没有数据库记录的受管容器

## 数据结构

### `challenge_runtime_configs`

保存题目级运行配置，包括：

- 镜像名
- 暴露端口
- 暴露协议
- TTL
- CPU / 内存限制
- 额外环境变量
- 启动命令覆盖

### `challenge_instances`

保存用户实例元数据，包括：

- 用户 ID
- 题目 ID
- 容器 ID / 容器名
- 映射端口
- 状态
- 启动时间
- 过期时间
- 停止时间

## 当前风险点

当前动态实例设计在比赛前仍需继续补齐以下问题：

1. 当前对账仍是基础版本，还没有区分更细的异常原因和告警级别
2. 需要增加按题目和按用户的配额与冷却策略
3. 需要为启动失败、停止失败和清理失败提供更明确的日志和错误分类
4. 需要把对账结果进一步接入结构化日志和监控指标

## 比赛前目标

动态实例模块在比赛前至少应达到以下状态：

- 数据库状态和容器真实状态一致
- 后台和选手侧的实例删除路径都能真正停止容器
- 能发现并清理启动失败或失联实例
- 能按题目设置较保守的资源和续期策略
- 能通过日志和指标快速定位实例层故障

## 当前明确不做的事

- 当前阶段不拆独立 runtime service
- 当前阶段不引入复杂调度系统
- 当前阶段不引入 Kubernetes 作为实例编排层

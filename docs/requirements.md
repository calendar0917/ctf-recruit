# 需求梳理

## 已确认决策

- 采用个人赛模式
- 优先支撑单场招新赛
- 动态实例作为 MVP 能力纳入
- 动态实例默认按用户独立分配
- 每个用户在同一道动态题上最多保留一个运行中实例
- 默认实例生命周期按 `30m` 设计，后续可改为题目级配置

## 核心用户故事

### 选手侧

- 作为选手，我可以注册账号并登录平台
- 作为选手，我可以查看比赛公告与开放时间
- 作为选手，我可以按分类浏览题目
- 作为选手，我可以下载题目附件
- 作为选手，我可以启动动态题目实例并获得访问地址
- 作为选手，我可以查看实例剩余时间
- 作为选手，我可以主动销毁自己的实例
- 作为选手，我可以提交 Flag 并得到反馈
- 作为选手，我可以查看自己的解题记录与总分
- 作为选手，我可以查看排行榜

### 管理员侧

- 作为管理员，我可以创建与编辑题目
- 作为管理员，我可以设置题目类型、分值、Flag、附件与运行配置
- 作为管理员，我可以发布公告
- 作为管理员，我可以开启、暂停、结束比赛
- 作为管理员，我可以查看所有提交记录
- 作为管理员，我可以封禁异常账号
- 作为管理员，我可以查看实例分配与运行状态

### 运维侧

- 作为运维，我可以配置题目容器镜像和启动方式
- 作为运维，我可以设置实例生命周期和资源限制
- 作为运维，我可以回收异常实例
- 作为运维，我可以查看服务日志和基础指标

## 非功能需求

### 安全性

- 密码必须安全存储，禁止明文
- Flag 不应在前端泄露
- 管理接口必须做权限控制
- 动态实例必须限制 CPU、内存、端口暴露和运行时间
- API 不应直接信任前端上报的实例状态
- 上传文件类型与大小需要限制
- 关键操作应保留审计日志

### 可用性

- 平台在比赛期间应保持稳定
- 关键页面加载时间应可接受
- 提交接口需要具备基础限流能力
- 动态实例失败时应返回明确错误信息
- 实例过期后应被后台任务自动回收

### 可维护性

- 前后端代码结构清晰
- 环境变量配置清楚
- 部署步骤可文档化复现
- 至少有基础单元测试和关键流程集成测试

## 约束与假设

- 初期部署环境按单机或单台云主机考虑
- 初期用户规模按几十到几百人考虑
- 动态题目数量有限，不做复杂调度系统
- 数据库优先选择关系型数据库
- 前台与管理后台先共用一个前端工程

## 暂未最终锁定的事项

- 是否需要邮件找回密码
- 附件是否首期上对象存储
- 动态实例是否支持手动续期
- 动态实例访问域名是统一网关还是端口映射

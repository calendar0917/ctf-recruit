# 需求梳理

## 已确认决策

- 采用个人赛模式
- 优先支撑单场校内招新赛，同时允许校外访问
- 动态实例作为首期能力纳入
- 动态实例默认按用户独立分配
- 每个用户在同一道动态题上最多保留一个运行中实例
- 当前认证方案采用 `用户名或邮箱 + 密码 + Bearer Token`
- 当前阶段保留单体 Go 架构，不做 reducer / worker 式重写

## 容量与运行假设

- 注册量目标：`500 ~ 1000`
- 峰值在线：按 `100 ~ 250` 在线用户留裕量
- 动态题数量有限，不做复杂调度系统
- 初期部署环境按单机或单台云主机考虑
- 比赛期间需要支持公网访问

## 核心用户故事

### 选手侧

- 作为选手，我可以注册并登录平台
- 作为选手，我可以查看公告、题目和附件
- 作为选手，我可以启动自己的动态题实例并获得访问地址
- 作为选手，我可以查看实例状态、剩余时间并续期或删除实例
- 作为选手，我可以提交 Flag 并得到反馈
- 作为选手，我可以查看自己的提交记录、解题记录和排行榜

### 管理员侧

- 作为管理员，我可以创建和编辑题目
- 作为管理员，我可以上传附件并设置动态运行配置
- 作为管理员，我可以发布和删除公告
- 作为管理员，我可以查看所有提交记录
- 作为管理员，我可以管理用户状态和角色
- 作为管理员，我可以查看并干预动态实例状态
- 作为管理员，我可以查看关键后台操作审计日志

### 运维侧

- 作为运维，我可以配置题目镜像、启动方式、TTL 和资源限制
- 作为运维，我可以回收异常实例并查看实例运行情况
- 作为运维，我可以查看服务健康状态、日志和基础指标
- 作为运维，我可以按文档完成部署、备份、恢复和赛前彩排

## 非功能需求

### 安全性

- 密码必须安全哈希存储，禁止明文
- Flag 不应在前端泄露
- 管理接口必须经过服务端权限控制
- 附件和题目访问必须经过可见性校验
- 动态实例必须限制 CPU、内存、端口暴露和运行时间
- 上传文件需要限制大小和基本类型
- 关键后台操作应记录审计日志
- 生产环境不得使用开发默认密钥或默认管理员口令

### 可用性

- 平台在比赛期间应保持稳定
- 登录、看题、提交和排行榜是最高优先级链路
- 动态实例失败时应返回明确错误信息
- 实例过期后应被后台任务自动回收
- API 重启后需要能够恢复关键状态并识别僵尸实例

### 可运维性

- 部署步骤必须可文档化复现
- 至少要有健康检查、readiness、日志和基础指标
- 必须有数据库备份和恢复方案
- 比赛前必须进行彩排和基础压测

### 可维护性

- 前后端代码结构清晰
- 环境变量和部署参数说明清楚
- 至少有基础单元测试和关键流程 smoke test
- 影响架构、接口和部署方式的改动必须同步更新文档

## 当前阶段暂不做的事

- 邮件系统和找回密码
- 复杂反作弊系统
- Writeup 平台
- 微服务拆分
- Kubernetes 调度
- 面向大型公开赛的复杂实时架构

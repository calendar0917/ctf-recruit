# 开发基线与升级路线

本文档是当前仓库后续开发的主基线文档。

用途：

- 固定项目接下来一段时间内的目标场景、容量假设和技术决策
- 作为功能取舍、架构演进和上线准备的统一依据
- 当其他文档与本文档冲突时，以本文档为准，并在同一个改动中同步修正文档分歧

## 目标场景

项目当前服务于单场校内 CTF 招新赛，同时允许校外访问。

当前阶段不追求把平台抽象成通用训练社区或大型公开赛通用底座，而是先把单场比赛的稳定性、运维性和安全性做到可上线水平。

## 目标规模

当前设计目标按以下量级推进：

- 注册量：`500 ~ 1000`
- 峰值在线：按 `100 ~ 250` 在线用户留裕量
- 动态题：数量有限，以少量高价值 Web / Pwn / Misc 题为主
- 动态实例：按“数十个同时运行实例”设计，具体上限由宿主机资源和题目配额共同决定

说明：

- 上述数字用于指导架构和压测目标，不是比赛当天的承诺值
- 若后续活动规模超出该范围，应先更新本文档，再讨论架构升级

## 核心架构决策

### 保留单体后端，不做 GeekGame 式重写

当前项目继续采用单体 Go API 作为主后端，不引入 reducer / worker、事件总线或消息队列作为核心依赖。

原因：

- 当前目标规模下，单体 Go + PostgreSQL 足以承载
- 当前主要风险来自运行时治理、部署、授权边界和观测缺口，而不是数据库或 HTTP 吞吐
- 过早引入复杂架构会增加维护和排障成本

### 数据库仍以 PostgreSQL 为唯一事实源

- 用户、题目、提交、实例、公告、审计日志均以 PostgreSQL 为主存储
- 排行榜和题目状态继续以数据库查询和应用层聚合为主
- 在比赛前阶段，不引入内存态主状态或复杂同步机制

### Redis 作为生产化增强，而不是核心状态依赖

- 开发环境可以不依赖 Redis 的核心能力
- 生产化阶段优先将 Redis 用于限流、短期缓存和轻量协调
- 不将 Redis 作为比赛核心数据的唯一存储位置

### 动态实例继续基于 Docker Engine，但要补治理

- 当前继续使用 Docker Engine 运行动态题实例
- 当前继续通过后端 runtime 模块调用 Docker API
- 在达到本文档目标规模之前，不拆独立 runtime service
- 当且仅当动态实例治理复杂度显著高于普通 API 时，再考虑把 runtime 模块独立出去

### 前后端仍保持单仓协作

- 选手端和管理端继续共用一个前端工程
- 优先保证一致的认证、权限、接口契约和发布流程
- 不在当前阶段拆多前端工程

## 当前项目基线

截至当前仓库状态，已具备以下能力：

- 注册、登录、Bearer Token 鉴权、`GET /me`
- 公告、题目列表、题目详情、附件下载、Flag 提交、排行榜
- 用户提交记录、解题记录查询
- 动态实例启动、查询、删除、续期、过期清理
- 管理端题目、附件、公告、提交记录、实例、用户、审计日志基础能力
- 基础健康检查、数据库连通性检查和后端单元测试

这说明项目已经不是“初始化脚手架”，而是一个可运行的单场比赛平台原型。

但它还没有达到“面向 500 ~ 1000 注册量稳定上线”的状态，主要差距在：

- 运行时一致性仍有缺口
- 部署链路仍偏开发态
- 授权和默认安全配置仍有明显短板
- 压测、观测、备份和演练能力不足

## 当前阶段的开发原则

### 原则 1：优先修正确性和上线阻塞问题

在比赛前阶段，优先级排序如下：

1. 安全与权限边界
2. 动态实例一致性和资源治理
3. 生产部署、观测和备份
4. 压测与演练
5. 新功能扩展

### 原则 2：不为了未来假想规模而提前重构

当前不做以下事项：

- 微服务拆分
- Kubernetes 调度
- reducer / worker 架构重写
- 消息队列驱动的核心流程
- 复杂反作弊系统
- 通用挑战 DSL 或重量级插件系统

### 原则 3：所有新增功能都要附带文档和验证

后续开发默认要求：

- 功能改动必须更新接口文档或基线文档
- 后端逻辑改动必须补对应单元测试
- 涉及部署和运维的改动必须补部署说明或运行手册
- 影响数据结构的改动必须提供迁移方案

## P1：上线前强烈建议完成的生产化增强

### P1-1 将限流从进程内内存升级到 Redis

状态：已完成基础版。

已完成：

- 登录、注册、Flag 提交和后台关键写接口已接入统一限流器
- 当 `REDIS_ADDR` 可用时，限流状态会写入 Redis，避免单实例重启后全部丢失
- Redis 不可用时会自动回退到进程内内存限流，并记录降级日志
- 开发与生产 Compose 已补齐 Redis 限流相关环境变量
- 前端已对登录、注册和后台写接口的限流错误给出明确提示

后续仍需关注：

- 当前 Redis 接入仍是最小实现，还没有暴露命中统计和剩余配额信息
- 后续应把限流命中率、降级次数和 Redis 连通性纳入监控指标
- 如果未来存在多入口网关或 CDN，需要重新评估按 IP 和标识组合的限流粒度

### P1-2 增加动态实例对账与僵尸清理

状态：已完成基础版。

已完成：

- API 启动时会先执行一次实例对账
- 定时任务会扫描数据库活跃实例与 Docker 受管容器并纠正明显不一致状态
- 数据库有记录但容器已丢失时，会终止该实例记录
- Docker 中存在无数据库记录的受管容器时，会执行清理

后续仍需关注：

- 对账结果当前主要靠日志，还没有结构化指标和告警
- 还需要进一步区分启动失败、停止失败、清理失败等异常类型

### P1-3 增加运行时配额与冷却策略

状态：已完成基础版。

已完成：

- `challenge_runtime_configs` 已支持 `max_active_instances` 和 `user_cooldown_seconds`
- 实例启动前会校验题目总并发实例上限
- 实例启动前会校验同一用户在同一题上的创建冷却
- 运行时接口已返回 `instance_capacity_reached` 和 `instance_cooldown_active` 两类明确错误码
- 管理端题目配置表单已支持维护上述两个字段

后续仍需关注：

- 当前配额仍是单题级基础版本，还没有做按资源池或题目分组的更细治理
- 还需要把配额拒绝和冷却命中纳入结构化日志与监控指标
- 若后续并发显著上升，需要评估数据库计数查询的索引与热点行为

### P1-4 建立基础观测和备份能力

状态：已完成基础版。

已完成：

- API 请求日志已切换为 JSON 结构化日志
- 已提供 `GET /api/v1/metrics` 基础指标出口
- 已覆盖基础 HTTP 请求计数、请求耗时累计、限流命中和限流错误计数
- 已提供数据库备份与恢复脚本及最小操作文档

后续仍需关注：

- 当前指标仍是最小版本，还没有数据库连接池、实例数量和后台任务成功率等更细指标
- 还需要把指标正式接入 Prometheus / Grafana 或等价采集链路
- 赛前必须完成一次真实备份恢复演练并记录结论

### P1-5 完成压测和容量验证

状态：已完成基础版。

已完成：

- 已提供 `tests/load/basic.py`，使用 `python3` 标准库即可执行基础压测
- 已覆盖公开读、登录态读、认证、Flag 提交和动态实例五类基线场景
- 已支持抓取 `/api/v1/metrics` 前后快照并输出请求统计、延迟分位和错误分布
- 已提供容量记录模板，支持赛前沉淀正式压测结论

后续仍需关注：

- 当前压测仍是单进程、单机、轻量模型，不等价于真实 `100 ~ 250` 活跃用户全量模拟
- 当前实例场景更偏运行时稳定性验证，还没有覆盖多账号并发抢占实例资源的压测模型
- 在接近正式宿主机规格的环境中，仍需完成至少一轮真实记录并据此修正限流和资源配置

## P2：比赛稳定后再推进的产品增强

### P2-1 比赛生命周期控制

目标：

目标：

- 明确 `admin`、`ops`、出题人等角色边界

## 里程碑定义

### 里程碑 A：内部可用

达成条件：

- P0-1 到 P0-4 完成
- 本地和测试环境可完整初始化
- 单元测试通过
- 管理端和选手端主链路可回归

### 里程碑 B：可彩排

达成条件：

- P0 全部完成
- P1-2、P1-4 至少完成基础版
- 已完成一次接近真实比赛流程的彩排
- 已完成一次基础压测并记录结论

### 里程碑 C：可上线

达成条件：

- P0 全部完成
- P1 核心项完成或有明确降级策略
- 有明确值班、告警、备份和回滚方案
- 比赛前进入冻结期，只接受阻塞级 bugfix

## 定义完成标准

后续任务默认按以下标准判断是否完成：

- 代码实现可运行
- 对应文档已更新
- 关键路径测试已补齐
- 需要的迁移、配置和部署说明已提供
- 已说明风险、回滚方式和已知限制

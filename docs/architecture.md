# 技术方案

本文档描述当前项目在 `500 ~ 1000` 注册量目标下采用的技术路线。

## 架构目标

当前架构目标不是追求通用化或超大规模，而是为单场校内招新赛提供：

- 足够稳定的选手主链路
- 可运维的动态题实例能力
- 清晰可控的部署和排障路径
- 与项目规模匹配的复杂度

## 核心决策

### 后端继续使用单体 Go API

- 语言：Go `1.26+`
- HTTP：标准库 `net/http`
- 数据访问：当前直接使用 PostgreSQL 驱动和自维护 repository 层
- 鉴权：Bearer Token

原因：

- 当前目标规模下，单体 Go 服务足以承载
- 主要风险点在动态实例治理、部署和观测，而不是 API 吞吐
- 保持简单架构更有利于比赛前排障和变更控制

### 前端继续使用单仓 React 应用

- Vite
- React
- TypeScript

当前继续让选手端和管理端共享同一个前端工程，优先保障接口契约、认证流和发布流程一致。

### PostgreSQL 是唯一事实源

- 用户、题目、提交、解题、公告、实例、审计均落在 PostgreSQL
- 比赛阶段不引入内存态主状态或复杂同步协议
- 需要提升性能时，优先做索引、查询优化和缓存，而不是先重写架构

### Redis 作为生产化增强引入

Redis 当前不是比赛核心状态依赖，后续优先用于：

- 注册 / 登录 / 提交限流
- 短期缓存
- 轻量协调状态

## 当前推荐架构

### 开发环境

```text
Browser
  -> Frontend Dev Server
  -> API
    -> PostgreSQL
    -> Redis(optional)
    -> Docker Engine
      -> Challenge Containers
```

### 当前生产骨架

```text
Browser
  -> Gateway Nginx
    -> Frontend Nginx (static files)
    -> Go API binary
      -> PostgreSQL
      -> Redis
      -> Docker Engine
        -> Challenge Containers
```

说明：

- 仓库已提供“静态前端 + 构建后的 Go 二进制 + 反向代理”最小生产骨架
- 网关与前端静态服务当前仍是两层 Nginx，后续可按部署复杂度再决定是否合并
- 当前不做 reducer / worker、多进程状态同步或消息队列化

## 动态实例设计原则

- 分配粒度：按 `用户 + 题目` 独立实例
- 默认限制：同一用户在同一题上最多一个运行中实例
- 访问方式：当前以宿主机端口映射为基线
- 生命周期：题目级 TTL，支持续期上限
- 清理方式：后台 sweeper + 后续补充启动时对账和僵尸清理
- 资源限制：CPU、内存、TTL 为必配项

## 安全基线

当前和后续实现都应满足以下要求：

- 密码哈希存储
- Token 必须有过期时间
- 管理接口必须经过服务端权限校验
- 文件上传必须限制大小和基础类型
- 动态容器不得使用 privileged 模式
- 动态容器必须限制 CPU、内存和生命周期
- 附件、题目和实例接口不得仅依赖前端控制可见性
- 生产初始化不得自动生成默认管理员口令

## 运维基线

比赛前必须补齐以下能力：

- 健康检查与 readiness 检查
- 构建产物部署，而不是源码运行
- 结构化日志和关键指标
- 数据库备份与恢复演练
- 动态实例失败、停止、清理的可观测性
- 最小化压测和赛前彩排

## 当前明确不做的事

- 微服务拆分
- Kubernetes 调度
- 消息队列驱动的核心流程
- GeekGame 式 reducer / worker 架构重写
- 复杂反作弊系统
- 多租户、多比赛并发运营平台
